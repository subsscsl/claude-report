Claude frequently misunderstands the scope or target of your requests, forcing you to intervene and redirect. Being more explicit upfront about targets, file locations, and the precise scope of changes—or using a CLAUDE.md with project conventions—could reduce these back-and-forth corrections.

  • Claude incorrectly moved Monitor items out of SKILL_ISSUES.md into the project checklist, misunderstanding your convention that systematic issues always stay in SKILL_ISSUES.md, forcing you to catch the error and have Claude revert.
UI/CSS Changes Requiring Multiple Revision Rounds
Visual implementation tasks consistently take more iterations than necessary because Claude struggles to translate your design intent from descriptions alone. Providing annotated screenshots, referencing specific CSS properties, or establishing a shared design token vocabulary upfront could significantly cut down on styling back-and-forth.
  • You had to clarify the tab padding/alignment request three separate times with screenshots before Claude achieved the correct full-width fill behavior, and even then the card implementation broke the tab styling.
  • Claude applied 40px top padding as you requested but it broke other role rows, requiring you to flag the regression and Claude to revert—a change that should have been scoped more carefully to avoid side effects.
Debugging Detours Down Wrong Root Causes
When diagnosing issues, Claude sometimes fixates on incorrect hypotheses (SSL errors, machine uptime, tool feasibility) before finding the real problem, costing you significant debugging time. Sharing more environmental context upfront—like your machine state, exact error output, or known working configurations—could help Claude skip the red herrings.
  • When previewing /sales/playground locally, Claude spent many rounds chasing SSL/cert/HSTS red herrings before discovering the real issue was a missing route and port conflict, turning a quick fix into an extended debugging session.
  • For the scheduled news digest failure, Claude initially assumed your machine wasn't on at noon and incorrectly stated launchd wasn't feasible—you had to correct both wrong assumptions before the actual launchd configuration issue was diagnosed.
